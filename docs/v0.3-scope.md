# v0.3 — Extensibility: Scope & Plan

This document scopes the v0.3 "Extensibility" milestone against the current
architecture. It is a planning artifact, not a spec set in stone.

## Status of the original roadmap items

The README listed four items. Two are **already done**:

- ✅ **Step `condition` evaluation** — `packages/engine/src/executor.ts` already
  evaluates `step.condition` and skips falsy steps; covered by the engine and
  integration tests.
- ✅ **`transform.json` with JSONPath expressions** — implemented via a
  self-contained, eval-free evaluator (`packages/connectors/src/jsonpath.ts`),
  wired through the connector's `expression` field.

Two features remain:

- ⬜ Plugin / custom connector system
- ⬜ Parallel step execution groups

---

## Feature A — Plugin / custom connector system · size: L

The engine is already plugin-ready: `step.type` is an open `z.string()` and the
executor does `connectors.get(step.type)` → throws `ConnectorNotFoundError` if
missing. So custom connectors already *run*; v0.3 is about **loading** them.

**Design**

1. **Project config file** — `autonode.config.json`, discovered by walking up
   from the cwd, with a `plugins: string[]` field (npm package names or local
   paths).
2. **Plugin contract** — a stable interface in a new `@autonode/plugin-sdk`:
   ```ts
   export interface AutonodePlugin {
     name: string;
     connectors?: Record<string, ConnectorFn>; // future: triggers, hooks
   }
   export default function activate(): AutonodePlugin { /* ... */ }
   ```
3. **Loader** — `buildRegistry(config)` starts from `createDefaultRegistry()`,
   dynamically `import()`s each plugin, validates its shape with zod, and merges
   connectors. Collision policy: error on duplicate type unless `override: true`.
4. **DRY refactor (prerequisite)** — `run`, `schedule`, and `webhook` each call
   `createDefaultRegistry()` independently. Extract one `buildRegistry()` in
   `apps/cli` that loads config + plugins, and route all three through it.
5. **CLI surface** — `autonode plugins list`; a `--no-plugins` escape hatch;
   `validate` may optionally warn on connector types absent from the registry.

**Risks** — plugins are arbitrary code execution (full trust); document the
trust model and ship the `--no-plugins` flag. Module resolution must target the
*project's* `node_modules`, not Autonode's.

**Files** — new config loader, plugin loader, SDK package; refactor 3 commands;
`init` scaffolds a config; `docs/plugins.md`; unit + integration tests.

## Feature B — Parallel step execution groups · size: M

The executor is a sequential `for` loop threading an immutable `currentContext`.

**Recommended first step — explicit groups.** A step carries
`parallel: StepDefinition[]`; the executor runs the children with `Promise.all`
against a shared context snapshot, then merges their results back into context.
Low risk, small schema change.

```yaml
- id: fanout
  parallel:
    - id: fetch_a
      type: http.request
      with: { url: "..." }
    - id: fetch_b
      type: http.request
      with: { url: "..." }
```

**Deferred — dependency DAG (`needs: [...]`).** More powerful but a real
executor rewrite plus topological scheduling; revisit in v0.4 if desired.

**Decisions to nail down** — fail-fast vs. `allSettled` collect; children cannot
see each other's outputs (document it); per-child retry/timeout/storage records
still apply.

**Files** — `StepDefinition` schema, executor, storage step-records,
`docs/parallel.md`, tests.
